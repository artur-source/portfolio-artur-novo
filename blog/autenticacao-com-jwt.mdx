## O Problema: Autenticação Escalável e Segura em Aplicações Distribuídas

Em aplicações web modernas, especialmente aquelas construídas com arquiteturas distribuídas (microserviços, APIs desacopladas, SPAs), a autenticação baseada em sessão tradicional pode se tornar um gargalo. Gerenciar o estado da sessão em múltiplos servidores, lidar com a replicação de sessões e garantir a escalabilidade horizontal de forma eficiente são desafios complexos. O problema central é como fornecer um mecanismo de autenticação **stateless**, **seguro** e **escalável** que permita que os clientes (frontends) se comuniquem com os recursos do backend sem a necessidade de um estado de sessão persistente no servidor.

Foi nesse contexto que o **JSON Web Token (JWT)** emergiu como uma solução robusta, amplamente adotada em projetos como Core & Patch e RegiFlex. A decisão de usar JWT não foi apenas por popularidade, mas pela sua capacidade de encapsular informações de autenticação e autorização de forma autocontida e verificável.

### Decisões de Arquitetura: JWT vs. Sessão Tradicional

A escolha entre JWT e sessões tradicionais depende do contexto da aplicação:

1.  **Sessão Tradicional:** O servidor cria uma sessão, armazena seu estado (ex: ID do usuário) e envia um cookie de sessão para o cliente. Cada requisição subsequente inclui o cookie, e o servidor verifica o estado da sessão. É stateful, o que pode dificultar a escalabilidade horizontal.
2.  **JWT:** Após a autenticação, o servidor gera um token que contém as informações do usuário (payload) e uma assinatura criptográfica. Este token é enviado ao cliente, que o armazena e o envia em cada requisição. O servidor verifica a assinatura do token para validar a autenticidade e integridade, sem precisar consultar um banco de dados de sessões. É stateless, ideal para APIs e microserviços.

Para nossos produtos, que visam ser **SaaS escaláveis** e com **APIs desacopladas**, o JWT foi a escolha natural. Ele simplifica a arquitetura do backend, elimina a necessidade de gerenciar sessões distribuídas e permite que diferentes serviços validem o token de forma independente.

### Estrutura de Access Token + Refresh Token

Uma implementação segura de JWT geralmente envolve dois tipos de tokens:

-   **Access Token:** Curta duração (ex: 15-60 minutos). Usado para autorizar o acesso a recursos protegidos. Se for comprometido, seu tempo de vida limitado minimiza o risco.
-   **Refresh Token:** Longa duração (ex: 7-30 dias). Usado para obter novos access tokens quando o atual expira. Deve ser armazenado de forma mais segura e ter mecanismos de revogação.

Essa estratégia de dois tokens equilibra segurança e usabilidade. O access token, por ser de curta duração, pode ser armazenado em memória (no frontend) ou em `localStorage` (com ressalvas, como veremos). O refresh token, por sua vez, exige um armazenamento mais seguro.

### Onde Armazenar com Segurança: `httpOnly` Cookie vs. `localStorage`

Esta é uma das decisões mais críticas e frequentemente mal compreendidas na implementação de JWT:

-   **`localStorage`:** **É um erro comum e perigoso armazenar o access token (e especialmente o refresh token) em `localStorage`**. Embora seja fácil de usar, `localStorage` é vulnerável a ataques de Cross-Site Scripting (XSS). Se um atacante conseguir injetar código malicioso na sua página, ele pode facilmente acessar e roubar os tokens armazenados em `localStorage`, comprometendo a sessão do usuário.
-   **`httpOnly` Cookie:** **A forma mais segura de armazenar o refresh token é em um cookie `httpOnly`**. Um cookie `httpOnly` não pode ser acessado via JavaScript, o que o torna imune a ataques XSS. Além disso, deve ser configurado com `Secure` (apenas HTTPS) e `SameSite=Lax` (ou `Strict`) para mitigar ataques CSRF (Cross-Site Request Forgery).

O **access token** de curta duração pode ser armazenado em memória no frontend (variável JavaScript) e enviado no cabeçalho `Authorization` das requisições. Quando expira, o frontend usa o refresh token (do `httpOnly` cookie) para solicitar um novo access token ao backend.

### Rotação de Refresh Token e Erros Comuns

Para aumentar a segurança, é recomendável implementar a **rotação de refresh tokens**. Cada vez que um refresh token é usado para gerar um novo access token, um **novo refresh token** também é gerado e o antigo é invalidado. Isso impede ataques de reuso de refresh token: se um refresh token for roubado e usado, o próximo uso do token antigo (pelo usuário legítimo ou pelo atacante) será detectado como inválido, indicando um comprometimento.

Um erro comum cometido por desenvolvedores juniores é **não invalidar o token no logout** ou definir um **tempo de expiração longo demais** para o access token. No logout, tanto o access token quanto o refresh token devem ser invalidados no servidor (se houver um mecanismo de lista negra ou banco de dados de tokens) e removidos do cliente. Um access token com tempo de vida muito longo aumenta a janela de oportunidade para um atacante em caso de roubo.

### Trecho de Código: Middleware de Verificação de JWT

Este exemplo em TypeScript para um backend Node.js ilustra um middleware básico para verificar um JWT e extrair as informações do usuário:

```typescript
import { Request, Response, NextFunction } from 'express';
import jwt from 'jsonwebtoken';

// Definição de tipo para o payload do JWT
interface JwtPayload {
  userId: string;
  tenantId?: string; // Opcional para aplicações multi-tenant
  // Outras informações do usuário
}

// Estender a interface Request do Express para incluir o usuário
declare global {
  namespace Express {
    interface Request {
      user?: JwtPayload;
    }
  }
}

const JWT_SECRET = process.env.JWT_SECRET || 'supersecretjwtkey'; // Usar variável de ambiente!

export const authenticateToken = (req: Request, res: Response, next: NextFunction) => {
  // 1. Obter o token do cabeçalho Authorization
  const authHeader = req.headers['authorization'];
  const token = authHeader && authHeader.split(' ')[1]; // Formato: Bearer TOKEN

  if (token == null) {
    return res.status(401).json({ message: 'Token não fornecido.' });
  }

  // 2. Verificar e decodificar o token
  jwt.verify(token, JWT_SECRET, (err, user) => {
    if (err) {
      // Erros comuns: Token expirado (TokenExpiredError), Token inválido (JsonWebTokenError)
      console.error('Erro na verificação do JWT:', err.message);
      return res.status(403).json({ message: 'Token inválido ou expirado.' });
    }

    // 3. Anexar o payload do usuário à requisição
    req.user = user as JwtPayload;
    next(); // Continuar para a próxima função middleware/rota
  });
};

// Exemplo de uso em uma rota protegida:
// app.get('/api/protected-data', authenticateToken, (req, res) => {
//   res.json({ data: `Olá, ${req.user?.userId}! Seus dados protegidos.` });
// });
```

Este middleware intercepta as requisições, extrai o JWT do cabeçalho `Authorization`, verifica sua validade usando a chave secreta (`JWT_SECRET`) e, se válido, anexa o payload decodificado (`userId`, `tenantId`, etc.) ao objeto `req.user`, tornando-o disponível para as rotas subsequentes. Erros como token ausente, inválido ou expirado são tratados com respostas HTTP apropriadas.

### Conclusão Prática: Segurança como Prioridade

A autenticação com JWT, quando implementada corretamente, oferece uma solução poderosa e escalável para aplicações modernas. As lições aprendidas em projetos como RegiFlex e Core & Patch reforçam que a segurança não é um recurso a ser adicionado no final, mas um pilar fundamental da arquitetura. Priorizar o armazenamento seguro de tokens (especialmente refresh tokens em `httpOnly` cookies), implementar rotação de tokens e tratar adequadamente os cenários de expiração e invalidação são práticas essenciais para proteger as aplicações e os dados dos usuários. Ignorar esses detalhes pode levar a vulnerabilidades sérias, transformando uma ferramenta poderosa em um vetor de ataque.
