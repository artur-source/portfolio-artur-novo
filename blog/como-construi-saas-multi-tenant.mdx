## O Problema: Escalar um SaaS para Múltiplas Clínicas com Segurança

Ao iniciar o desenvolvimento do RegiFlex, um sistema de gestão para clínicas, o desafio central não era apenas construir um software funcional, mas criar uma **plataforma SaaS multi-tenant** que pudesse atender a diversas clínicas de forma independente, garantindo **isolamento de dados rigoroso** e **escalabilidade** sem comprometer a segurança. A tentação inicial de um banco de dados único com um `tenant_id` em cada tabela era forte pela simplicidade aparente, mas as implicações de segurança e complexidade de manutenção em longo prazo eram preocupantes.

### Decisões de Arquitetura: Schema-per-Tenant vs. Row-Level Security

A escolha entre diferentes estratégias de multi-tenancy é crucial. Avaliamos duas abordagens principais:

1.  **Schema-per-Tenant:** Cada cliente possui seu próprio schema (ou até mesmo banco de dados) isolado. Isso oferece o maior nível de isolamento de dados, mas aumenta a complexidade de gerenciamento de banco de dados e deploy.
2.  **Row-Level Security (RLS) com `tenant_id`:** Todos os clientes compartilham o mesmo banco de dados e tabelas, mas cada registro é marcado com um `tenant_id`. A segurança é garantida por políticas de RLS no banco de dados, que filtram automaticamente os dados visíveis para cada tenant com base no `tenant_id` do usuário logado.

Para o RegiFlex, optamos pela abordagem de **Row-Level Security (RLS) com `tenant_id`**. A principal razão foi o **equilíbrio entre isolamento de dados e simplicidade operacional**. Gerenciar múltiplos schemas ou bancos de dados para cada cliente se tornaria um gargalo à medida que o número de clínicas crescesse. Com RLS, a lógica de filtragem é delegada ao próprio PostgreSQL, garantindo que, mesmo que um desenvolvedor cometa um erro na aplicação, o banco de dados impede o acesso a dados de outros tenants.

### tRPC e Drizzle ORM: Garantindo a Segurança em Nível de Tipo

A combinação de **tRPC** e **Drizzle ORM** com TypeScript foi fundamental para reforçar a segurança e a robustez do sistema. O tRPC nos permitiu construir uma API type-safe, onde os contratos de dados são validados em tempo de compilação, reduzindo drasticamente a chance de erros de comunicação entre frontend e backend. O Drizzle ORM, por sua vez, oferece uma interface type-safe para interagir com o banco de dados, permitindo que as queries sejam construídas de forma segura e validada pelo TypeScript.

Um dos maiores benefícios dessa combinação é a capacidade de **evitar vazamento de dados entre tenants em nível de tipo**. Ao definir os schemas do Drizzle com o `tenant_id` como uma chave estrangeira e aplicar as políticas de RLS no PostgreSQL, garantimos que qualquer tentativa de acessar dados sem o `tenant_id` correto resultaria em um erro de tipo ou em uma query que não retornaria dados, protegendo contra acessos indevidos.

### Autenticação e Autorização por Tenant

A autenticação no RegiFlex é gerenciada pelo Supabase Auth, que se integra perfeitamente com o PostgreSQL e RLS. Após o login, o `tenant_id` do usuário é extraído e utilizado para definir a sessão do banco de dados, garantindo que todas as operações subsequentes sejam automaticamente filtradas pelo RLS. A autorização é feita através de roles e permissões, também gerenciadas pelo Supabase, permitindo um controle granular sobre o que cada usuário pode fazer dentro de seu respectivo tenant.

### Trecho de Código: Query com Isolamento por `tenant_id`

Para ilustrar como o isolamento por `tenant_id` funciona em uma query, considere o seguinte exemplo usando Drizzle ORM e PostgreSQL com RLS:

```typescript
// Esquema do Drizzle para a tabela 'pacientes'
import { pgTable, uuid, text, timestamp } from 'drizzle-orm/pg-core';

export const patients = pgTable('patients', {
  id: uuid('id').defaultRandom().primaryKey(),
  tenantId: uuid('tenant_id').notNull(), // Chave estrangeira para o tenant
  name: text('name').notNull(),
  email: text('email').notNull().unique(),
  createdAt: timestamp('created_at').defaultNow(),
});

// Exemplo de política RLS no PostgreSQL (simplificado)
/*
CREATE POLICY tenant_isolation_policy ON patients
FOR ALL
USING (tenant_id = current_setting('app.tenant_id')::uuid);

ALTER TABLE patients ENABLE ROW LEVEL SECURITY;
*/

// Função no backend para buscar pacientes de um tenant específico
import { db } from './db'; // Sua instância do Drizzle
import { patients } from './schema';
import { eq } from 'drizzle-orm';

interface GetPatientsByTenantParams {
  tenantId: string;
}

export const getPatientsByTenant = async ({ tenantId }: GetPatientsByTenantParams) => {
  // Em um ambiente real, o tenantId seria definido na sessão do banco de dados
  // antes da execução da query, e o RLS faria o filtro automaticamente.
  // Aqui, para fins de demonstração, incluímos explicitamente na query.
  const result = await db.select().from(patients).where(eq(patients.tenantId, tenantId));
  return result;
};

// Exemplo de uso (no seu tRPC router, por exemplo)
// const patients = await getPatientsByTenant({ tenantId: ctx.user.tenantId });
```

Neste exemplo, a coluna `tenantId` na tabela `patients` é crucial. A política de RLS no PostgreSQL garantiria que, mesmo que a cláusula `WHERE` fosse omitida acidentalmente na aplicação, o banco de dados só retornaria os pacientes associados ao `tenant_id` da sessão atual. O Drizzle, com sua tipagem forte, ajuda a garantir que o `tenantId` seja sempre considerado nas operações.

### Conclusão Prática: O Que Faria Diferente Hoje?

Se o RegiFlex triplicasse de clientes amanhã, a principal área de otimização seria a **estratégia de sharding de banco de dados**. Embora o RLS seja excelente para isolamento lógico e escalabilidade vertical, um crescimento exponencial poderia levar a gargalos de performance em um único servidor de banco de dados. Nesse cenário, eu exploraria:

-   **Sharding Horizontal:** Distribuir os dados de diferentes tenants em múltiplos servidores de banco de dados, com base no `tenant_id`. Isso permitiria escalabilidade horizontal e maior resiliência.
-   **Pool de Conexões Otimizado:** Implementar um pool de conexões mais sofisticado para o banco de dados, capaz de gerenciar eficientemente as conexões por tenant e reduzir a latência.
-   **Cache Distribuído:** Utilizar soluções de cache distribuído (e.g., Redis) para dados frequentemente acessados, reduzindo a carga sobre o banco de dados.

A escolha do RLS com `tenant_id` foi um trade-off consciente, priorizando a agilidade no desenvolvimento e a segurança inicial. Para um crescimento massivo, a evolução para sharding seria o próximo passo lógico, mantendo a base de RLS para a segurança granular dentro de cada shard. Essa abordagem iterativa permite que a arquitetura evolua com as necessidades do negócio, sem super-engenharia inicial desnecessária.
